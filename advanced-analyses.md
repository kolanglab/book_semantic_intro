# 発展的・最先端の意味解析

ここまでで、名前解決から抽象解釈まで、意味解析の幹を一通り歩いてきました。最後の章では、その幹から伸びた研究の最前線を一望します。共通するテーマは、「型やフロー解析を**もっと強く**して、これまで実行時にしか分からなかった性質を、コンパイル時に保証する」ことです。難しい理論も登場しますが、どれも前章までの考え方の延長線上にあります。「何の動機で、何を保証し、どんな手法か」を軸に読んでください。

## 篩型：型に述語を載せる

第2部の型は「`Int`」「`String`」のように値の**種類**しか語れませんでした。でも本当は「**正の** `Int`」「**空でない**配列」「**0でない**除数」と言いたい場面が多い。型に論理的な条件（述語）を貼り付けて表現力を上げたものが **篩型（refinement type, 篩い型）** です。たとえば、

```text
{ v: Int | v > 0 }          正の整数の型
{ v: Int | v ≠ 0 }          ゼロでない整数の型
div : Int -> { v: Int | v ≠ 0 } -> Int   ゼロ除算が型で防がれる割り算
```

`div` の第2引数の型が `{ v: Int | v ≠ 0 }` なので、ゼロを渡すコードは**型エラー**になります。ゼロ除算という実行時エラーが、型検査でコンパイル時に弾かれるのです。正確に言えば、受理されるのは型検査器が「除数はゼロでない」と**証明できた**呼び出しだけです。証明できない呼び出しは ── たとえ実行時には決してゼロにならないとしても ── 安全側に倒して拒否されるため、実際には追加の注釈や条件の補強が必要になることもあります。

問題は、こうした条件付きの型をどう**検査**するかです。`x` が `{v | v > 0}` 型で `y` が `{v | v > 0}` 型のとき、`x + y` が `{v | v > 0}` 型だと示すには、「`x > 0` かつ `y > 0` ならば `x + y > 0`」という**論理式の正しさ**を判定しなければなりません。これは型検査というより**定理証明**です。

ここで活躍するのが **SMT ソルバ（SMT solver）** です。SMT は *Satisfiability Modulo Theories*（理論付き充足可能性）の略で、「整数の不等式」や「配列」などの理論を組み込んだうえで、論理式が**充足可能か**（式を成り立たせる値の割り当てが存在するか）を自動判定するエンジンです。「ある前提から結論が必ず従う」ことを示したいときは、その**否定**が充足不能であることを確かめる、という形で使います。Microsoft Research の **Z3** がその代表で[](#cite:demoura2008)、いまや検証技術の縁の下の力持ちになっています。篩型の検査は、型の条件から論理式を生成し、それを SMT ソルバに投げて自動的に証明することで実現されます。

この路線を実用レベルに引き上げたのが、Rondon・Kawaguchi・Jhala の **Liquid Types** です[](#cite:rondon2008)。彼らの鍵となるアイデアは、第2部の **ヒンドリー・ミルナー型推論**（型変数に制約を集めて解く）と、**述語の自動推論**（あらかじめ用意した述語の組み合わせから当てはまるものを探す）を組み合わせ、**篩型の大部分を自動で推論する**ことでした。つまり「正である」「範囲内である」といった注釈をプログラマがほとんど書かなくても、処理系が SMT ソルバの力を借りて導いてくれる。型推論の素直な延長が、強力な検証へと化けた好例です。今日では LiquidHaskell や F\* などがこの系譜にあります。

> [!NOTE]
> 篩型は、第4部冒頭の抽象解釈とも縁続きです。「述語の集まり」を抽象領域とみなせば、述語の推論はその領域での不動点計算になります。型・抽象解釈・定理証明という、別々に育った3つの分野がここで合流しているのです。

## エフェクトシステム：副作用を型で追う

型は「この式がどんな**値**を返すか」を語ります。では「この式が実行されると、値を返す以外に**何が起きるか**」── ファイルに書き込む、例外を投げる、状態を書き換える、といった **副作用（side effect）** ── も型で語れないでしょうか。それを可能にするのが **エフェクトシステム（effect system, 作用システム）** です。

エフェクトシステムでは、式の判定を「型 `T`」だけでなく「型 `T` と作用の集合 `E`」の組で行います。たとえば、

```text
readFile  : Path -> String  &  { IO }       ファイル入出力という作用を持つ
pureAdd   : Int -> Int -> Int  &  { }        作用なし（純粋）
mayThrow  : Int -> Int  &  { Exn }           例外を投げうる
```

`&` の右側がその関数の**作用**です。これがあると、「純粋なはずの関数の中で、こっそり IO をしていないか」「捕捉していない例外を投げる可能性はないか」をコンパイル時に追跡できます。この枠組みを多相に拡張し、作用についても型変数のように一般化できるようにしたのが、Lucassen と Gifford の **多相エフェクトシステム** です[](#cite:lucassen1988)。

エフェクトの考え方は近年、**代数的効果（algebraic effects）** という形で実装言語に入りつつあります。さらに身近な例として、Java の検査例外（`throws` 宣言）は素朴なエフェクトシステムの一種です。Koka は、作用を型に明示して静的に追跡する言語の代表例です。一方、OCaml 5 のエフェクトハンドラは代数的効果を**言語機能**として導入したもので、Koka のような静的なエフェクト型を標準で備えているわけではありません。計算モデルとしての代数的効果と、それを型で追跡するエフェクトシステムは、関連はしますが同一ではない点に注意してください。「値だけでなく**振る舞い**も型で約束する」という発想は、いま最も活発に研究されている方向の一つです。

## リージョンと所有権：メモリ安全性を型で保証する

C や C++ で最も悩ましいバグは、メモリ管理にまつわるもの ── 解放済みメモリの使用（use-after-free）、二重解放、メモリリーク ── です。ガベージコレクション（GC）はこれらを実行時に防ぎますが、実行時コストがかかります。「GC なしで、しかもコンパイル時にメモリ安全を保証できないか」という問いが、この分野を駆動してきました。

最初の大きな一歩が、Tofte と Talpin の **リージョンに基づくメモリ管理（region-based memory management）** です[](#cite:tofte1997)。メモリを **リージョン（region, 領域）** という単位でまとめて確保・解放し、「どの値がどのリージョンに属し、いつまで生きるか」を**型と作用の解析で自動推論**する、というアイデアです。第2部の型推論と、本章のエフェクトシステムを組み合わせて、メモリの寿命を静的に決めてしまうわけです。リージョン推論がうまく働く範囲では、これにより GC なしでも安全な自動解放が実現できます。ただし、リージョンの粒度が粗いとメモリを長く抱え込みすぎるなど、推論の精度・注釈の量・他の言語機能との相互作用が実用上の課題で、GC の単純な完全置き換えになるわけではありません。

この系譜の現代的な到達点が、**Rust の所有権と借用検査（ownership and borrow checking）** です。Rust は、

- **所有権（ownership）**：各値の持ち主はただ1つ。持ち主がスコープを抜けたら値は解放される。
- **借用（borrowing）**：値への参照は「共有参照（読み取り専用、複数可）」か「可変参照（書き込み可、ただし1つだけ）」のどちらか。

という規則を**型システムで強制**します。これにより、データ競合や use-after-free が**コンパイル時に**排除されます。借用検査は、本質的には第3部の制御フロー/データフロー解析（参照の生存期間を CFG 上で追う、いわゆる NLL: non-lexical lifetimes）と、リージョン/所有権の型理論の融合です。

Rust の安全性が**本当に**正しいのかは、それ自体が大きな研究課題でした。Jung らの **RustBelt** プロジェクトは、Rust の中核部分について、`unsafe` を含むライブラリも込みで、メモリ安全性を**機械検証された形式的証明**として与えました[](#cite:jung2018)。「実用言語の安全性を数学的に証明する」という、理論と実装が手を取り合った象徴的な成果です。

> [!TIP]
> リージョン・所有権・借用は別々の概念に見えますが、根は同じ「**値の寿命（lifetime）を静的に追う**」です。GC が実行時に行っていた「もう誰も使わないメモリの発見」を、型・フロー解析でコンパイル時に前倒しした、と捉えると全体像が掴めます。

## 分離論理：ポインタを扱う論理

メモリ安全性を**論理**の側から攻めたのが、Reynolds の **分離論理（separation logic）** です[](#cite:reynolds2002)。プログラムの正しさを論理式で記述する古典的な枠組み（ホーア論理）は、ポインタや共有された可変データを扱うのが苦手でした。「`p` を書き換えたとき、`q` の指す先は変わるのか変わらないのか」を、いちいち全部のポインタについて場合分けしないと書けなかったのです。

分離論理の発明は、**分離積（separating conjunction）** という新しい結合子 `P * Q` です。これは「メモリを互いに**重ならない**2つの部分に分け、一方で `P`、もう一方で `Q` が成り立つ」と読みます。重ならないことが保証されているので、「`p` の領域を書き換えても、`q` の領域はそのまま」と**ローカルに**推論できます。ポインタ操作の証明が、関係する小さな領域だけを見て進められるようになったのです。

この「局所推論（local reasoning）」の力は絶大で、分離論理は連結リストや木のような**動的データ構造を持つプログラムの検証**を現実的なものにしました。次節で見る大規模ツールの理論的な土台にもなっています。

## 最前線を現場へ：大規模実用化

理論がどれほど美しくても、何百万行ものコードベースで毎日動かなければ、現場のバグは減りません。意味解析の最前線のもう一つの軸は、**実用規模へのスケール**です。

その代表が、Facebook（Meta）の静的解析ツール **Infer** です[](#cite:calcagno2015)。Infer は前節の**分離論理**を土台に、ヌルポインタ参照・メモリリーク・リソースリークなどを検出します。技術的な肝は2つです。第一に、関数ごとに「この関数が成り立つために必要な前提と、実行後に成り立つ結果」の組（**関数まとめ／summary**）を自動で導く **双方向解析（bi-abduction）** によって、第3部で触れた手続き間解析[](#cite:reps1995)を巨大なコードへスケールさせたこと。第二に、コード**全体**を毎回解析し直すのではなく、変更されたコミットの**差分だけ**を高速に解析し、開発者のプルリクエストにその場でコメントを返す、というワークフローへの統合です。

ここに、本書を貫くトレードオフの最終形が現れます。Infer は意図的に**健全性をあきらめています**（見逃しはある）。その代わり、報告するバグの確からしさと解析速度を最優先し、「開発者が無視しない量の、本物のバグ」を毎日のコードレビューに乗せることを選びました。学術的な完全性より、現場で使われ続けることを選んだ設計判断です。健全性・精度・速度・使いやすさのどれを取りどれを捨てるか ── 第1章で立てたこの問いに、現場は現場なりの答えを出している、というわけです。

> [!IMPORTANT]
> 最先端の手法は「より強い保証」と「より広い適用」の二方向に進んでいます。RustBelt や篩型のように**保証を極限まで強める**方向と、Infer のように**実用規模で日々回す**方向です。どちらが正しいということはなく、対象（人命に関わる制御ソフトか、高速に進化する Web サービスか）に応じて使い分けられています。意味解析に「唯一の正解」はありません。

## 結び ── AST のその先で

本書は、AST が与えられたところから出発しました。名前を解決し、型を検査・推論し、制御と値の流れを追い、抽象解釈という統一理論にたどり着き、そして篩型・所有権・分離論理という最前線を覗きました。

通して見えてきたのは、意味解析とは結局「**AST に書かれていない事実を、健全な近似によって導き出す**」営みであり、そのすべてが「動機（なぜ）・結果（何が分かる）・手法（どうやる）」という同じ骨格で語れる、ということです。そして本書が繰り返し立ち返ったのは、**健全性・精度・速度・使いやすさのトレードオフ**でした。どの解析も、このバランスのどこかに自分の場所を見つけています。

意味解析の研究は、いまも速く動いています。本書で身につけた地図 ── 名前・型・フロー・抽象解釈という4つの大陸 ── を手に、新しい論文やツールに出会ったときは、ぜひ「これはどの大陸の、どんなトレードオフの上に立っているのか」と問うてみてください。AST のその先には、まだ描き切れていない地図が広がっています。
