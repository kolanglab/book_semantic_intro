# 意味解析へようこそ

言語処理系（コンパイラやインタプリタ）は、おおまかに **フロントエンド** と **バックエンド** に分かれます。フロントエンドはソースコードを読み取って「理解」する側、バックエンドはその理解をもとに実行したり機械語を吐いたりする側です。本書が扱う **意味解析（semantic analysis）** は、ちょうどこの境目に立つ工程です。

この章では、意味解析とは何か、それが処理系のどこに位置するのか、そして本書で紹介する解析たちがどんな地図の上に並んでいるのかを、まず鳥の目で見渡します。

## 構文解析の次に来るもの

ソースコードが処理系に入ってから実行されるまでの典型的な流れは、次のようになっています。

```mermaid
graph LR
    A[ソースコード] --> B[字句解析]
    B --> C[構文解析]
    C --> D[AST]
    D --> E[意味解析]
    E --> F[中間表現 IR]
    F --> G[最適化・コード生成・実行]
```

**字句解析（lexing）** が文字列をトークン（単語）に区切り、**構文解析（parsing）** がトークン列を **抽象構文木（Abstract Syntax Tree, AST）** という木構造に組み上げます。AST は、プログラムの構文的な構造 ── どの式がどの式の中に入っているか ── を表した木です。本書はこの **AST が手に入ったところから** 話を始めます。構文解析そのものに興味がある読者は、姉妹編『構文解析入門』を参照してください。

問題は、AST ができただけではプログラムについてほとんど何も「分かっていない」ことです。たとえば次の Ruby 風のコードを考えます。

```ruby
def area(w, h)
  w * h
end

total = area(3, 4) + offset
```

構文解析が終わった時点で分かっているのは、「`area` という関数定義があり、`total` への代入があり、その右辺は関数呼び出しと `offset` の足し算だ」という**形**だけです。しかし、

- `offset` という名前はどこで定義されている？ そもそも定義されている？
- `area(3, 4)` の `+ offset` は、数同士の足し算として意味をなす？
- `area` は本当に2引数の関数？ 引数の個数は合っている？

といった問いには、まだ答えられません。これらはすべて、構文（形）ではなく **意味（中身）** に関する問いです。AST をさらに調べ、こうした問いに答え、プログラムが「意味として筋が通っているか」を確かめる工程 ── それが意味解析です。ドラゴンブックの名で知られる定番教科書[Aho et al., 2006](#cite:aho2006)も、コンパイラの章立てをこの「構文解析のあとの意味解析」を軸に構成しています。

## 意味解析は何のためにあるのか

意味解析の動機は、大きく3つに整理できます。本書を通じて、どの解析がどの動機に対応するのかを意識してください。

1. **誤りを早く見つける（検証）**：未定義の変数、型の合わない演算、引数の個数違いなどを、プログラムを実行する**前に**指摘する。バグの発見が早ければ早いほど、修正は安く済みます。
2. **実行を可能にする（情報の付与）**：「この `+` は整数加算か文字列連結か」「この変数はスタックのどこに置くか」といった、実行やコード生成に必要な情報を AST に書き込む。バックエンドはこの情報がないと動けません。
3. **より良くする（最適化）**：「この計算結果は使われない」「この変数は常に定数」といった事実を見つけ、無駄を削ったり高速化したりする材料を集める。

たとえば型検査（第2部）は主に (1) と (2) のため、データフロー解析（第3部）は主に (3) のため、というように、解析ごとに重心は違います。しかし「AST を調べて、そこに**書かれていない事実**を導き出す」という点はすべて共通です。

> [!NOTE]
> 意味解析の結果は「OK / NG の判定」だけではありません。多くの場合、AST の各ノードに**注釈（annotation）** を付け足します。「このノードの型は `Integer`」「この変数の格納先はローカル変数の2番目」といった具合です。注釈の付いた木を、しばしば **装飾構文木（decorated/annotated AST）** と呼びます。

## 静的か、動的か

意味解析には大きく2つの立場があります。プログラムを**実行せずに**ソース（AST）だけから調べる **静的解析（static analysis）** と、実際に**実行しながら**調べる **動的解析（dynamic analysis）** です。

- 静的解析の例：C コンパイラの型検査、未使用変数の警告、リンタの指摘。
- 動的解析の例：実行時の型チェック（Ruby の `NoMethodError` など）、サニタイザによるメモリエラー検出、プロファイラ。

本書が主に扱うのは静的解析です。なぜなら、静的解析こそ「AST を入力に取り、実行前に意味を調べる」という意味解析の王道だからです。ただし、静的解析には**原理的な限界**があることを最初に断っておきます。

> [!IMPORTANT]
> 「このプログラムは絶対にエラーで止まらないか？」のような、プログラムの実行時の振る舞いに関する非自明な性質は、一般には**判定不能（undecidable）** です。これは **ライスの定理（Rice's theorem）** として知られる計算理論の結果で、停止性問題が解けないことの帰結です。つまり、どんな静的解析も「完璧」にはなりえません。

この限界を受け入れたうえで、静的解析は2つの方向に妥協します。どちらに倒すかは設計判断です。

- **健全（sound）だが不完全**：「問題なし」と言ったら本当に問題ない。ただし安全側に倒すので、本当は問題ないコードまで「あやしい」と言うことがある（**偽陽性／false positive**）。型検査はこちら寄りです。
- **完全に近いが健全でない**：見つけた問題は本物だが、見逃し（**偽陰性／false negative**）がありうる。実用バグ検出ツールはしばしばこちら寄りです。

この「健全性」と「精度」のトレードオフは本書の通奏低音です。各解析を見るときは「これはどちら側に倒しているのか」を考えてみてください。

## 意味解析を支える共通の道具

個々の解析に入る前に、ほぼすべての意味解析が共通して使う考え方を2つだけ紹介します。

### 木をたどる：トラバーサルとビジターパターン

意味解析は結局のところ「AST をたどって、各ノードで何かを調べる・記録する」ことの繰り返しです。木を系統的にたどる処理を **トラバーサル（traversal）** と呼び、その典型的な実装が **ビジターパターン（visitor pattern）** です。ノードの種類ごとに「訪問時の処理」を定義し、木を再帰的にめぐります。本書のコード例でも繰り返し登場するので、Ruby で骨格を示しておきます。

```ruby
# AST ノードの種類ごとに visit_xxx を呼び分ける土台
class Visitor
  def visit(node)
    method_name = "visit_#{node.type}"   # 例: visit_binop, visit_var
    send(method_name, node)
  end

  # 子ノードを順にたどるためのヘルパ
  def visit_children(node)
    node.children.each { |c| visit(c) }
  end
end
```

具体的な解析は、この `Visitor` を継承し、`visit_binop`（二項演算）や `visit_var`（変数参照）といったメソッドを埋めることで作ります。「どのノードで何をするか」だけに集中でき、木の歩き方は使い回せる、というのが利点です。

### 形式的な仕様：属性文法

「各ノードで何を計算するか」をきちんと**仕様**として書きたくなります。そのための古典的な枠組みが、Knuth が1968年に提案した **属性文法（attribute grammar）** です[Knuth, 1968](#cite:knuth1968)。属性文法では、文法の各記号（ノード種別）に **属性（attribute）** という値を結びつけ、その値の計算規則を、隣り合うノードの属性を使った式として記述します。

属性には2種類あります。

- **合成属性（synthesized attribute）**：子の属性から親の属性を計算する（情報が木の**下から上**へ流れる）。たとえば「式の型」は典型的な合成属性です。子（部分式）の型が決まれば、親（全体の式）の型が決まります。
- **継承属性（inherited attribute）**：親（や兄弟）の属性から子の属性を計算する（情報が**上から下**へ流れる）。たとえば「いま有効なスコープ（使える名前の一覧）」は継承属性です。外側のブロックから内側へ受け渡されます。

たとえば「定数式の値を計算する」属性 `val` は、次のような規則で書けます（`E` は式、`→` は文法規則）。

```text
E → E1 + E2    {  E.val = E1.val + E2.val  }   ← 合成属性
E → 数値リテラル {  E.val = 数値リテラルの値      }
```

属性文法は「意味解析とは AST 上での属性の計算である」という見方を与えてくれます。実際の処理系が属性文法の記法をそのまま使うとは限りませんが、**合成属性（下から上）と継承属性（上から下）** という2方向の情報の流れは、このあとのすべての章に顔を出します。型は下から、スコープは上から ── このイメージを持っておくと、各解析の見通しが一気に良くなります。

> [!TIP]
> 「下から上か、上から下か」を意識すると、解析の実装がぐっと書きやすくなります。子を先に処理してから親をまとめるなら合成属性的（帰りがけ）、親で決めた情報を子に渡すなら継承属性的（行きがけ）です。ビジターを書くときの設計指針になります。

## 本書の地図

最後に、これから訪れる解析たちを一枚の地図にまとめます。簡単なもの（上）から最先端（下）へ、おおよそ難易度順に並んでいます。

| 部 | 章 | 解析 | 主な動機 | 代表的な結果 |
|----|----|------|----------|--------------|
| 1 | 名前解決 | 名前解決・スコープ解析 | 検証・情報付与 | 各名前の定義の確定 |
| 2 | 型検査 | 型検査 | 検証・情報付与 | 型の整合性・各式の型 |
| 2 | 型推論 | 型推論（HM法） | 情報付与 | 注釈なしでの型の決定 |
| 3 | 制御フロー | 制御フロー解析 | 最適化・検証 | 実行経路（CFG） |
| 3 | データフロー | データフロー解析 | 最適化 | 到達定義・生存変数など |
| 4 | 抽象解釈 | 抽象解釈 | （統一理論） | 健全な近似の枠組み |
| 4 | 発展 | 篩型・所有権・分離論理ほか | 検証 | 強い安全性保証 |

それぞれの章は、**動機 → 結果 → 手法 → 具体例** の順に進みます。では第1部、すべての意味解析の出発点である「名前解決」から始めましょう。
