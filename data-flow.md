# データフロー解析

CFG という「実行経路の地図」ができました。次に知りたいのは、その地図の上を**値や情報がどう流れるか**です。「この変数の値はここまで生き残る？」「この計算結果はどこで使われる？」「この変数は常に定数か？」── こうした問いに、プログラムを実行せずに答えるのが **データフロー解析（data flow analysis）** です。コンパイラ最適化の土台であり、静的解析の王道でもあります。

## 動機：使われ方を知れば無駄が見える

最適化の多くは「無駄を見つけて削る」ことに尽きます。そして無駄を見つけるには、値の流れを知る必要があります。代表的な解析と、それが可能にする最適化を挙げます。

- **生存変数解析（live variable analysis）**：「ある時点から先で、その変数の値が（再代入される前に）使われることがあるか」を調べる。使われない変数への代入は**デッドコード**として削除でき、レジスタ割り当ての基礎にもなる。
- **到達定義解析（reaching definitions）**：「ある地点に届きうる代入はどれか」を調べる。定数伝播や、変数の使用元をたどる解析に使う。
- **定数伝播（constant propagation）**：「ある変数がその地点で常に同じ定数か」を調べ、定数ならその場で計算済みの値に置き換える。

いずれも **動機は最適化**（と、それに付随するバグ検出）です。たとえば「`x` に代入したのに、その後 `x` が一度も読まれずに別の値で上書きされる」なら、最初の代入は消してよい ── これを正しく行うには生存変数解析が要ります。

## すべては「ある情報が、各地点で何か」を求める問題

一見ばらばらに見えるこれらの解析は、実は**同じ枠組み**に収まります。共通の問いはこうです ── 「CFG の各地点（各ブロックの入口・出口）で、ある種類の情報はどうなっているか」。これを求めるために、3つの部品を決めます。

1. **何を追うか（データフロー値）**：生存変数なら「生きている変数の集合」、到達定義なら「届いている代入の集合」。
2. **各ブロックが情報をどう変えるか（伝達関数, transfer function）**：ブロックを通ると情報がどう書き換わるか。
3. **分岐や合流で情報をどう束ねるか（合流演算, meet/join）**：複数の経路が合流する点で、情報をどう一つにまとめるか（集合の和か、積か）。

この枠組みを、生存変数解析を例に具体化します。生存変数は CFG を**後ろ向き**（出口から入口へ）にたどって解く解析です。各ブロック `B` について、

```text
  in[B]  = use[B] ∪ (out[B] − def[B])
  out[B] = ∪ { in[S] | S は B の後続ブロック }
```

という連立方程式を立てます。記号の意味は、

- `use[B]`：`B` の中で、再代入される前に**読まれる**変数（＝ `B` に入る時点で生きていてほしい変数）。
- `def[B]`：`B` の中で**代入される**変数。
- `in[B]` / `out[B]`：`B` の入口／出口で生きている変数の集合。

`out[B] = ∪ in[S]` は「後続のどれかで生きているなら、`B` の出口でも生きている」という合流（ここでは**和集合**）。`in[B] = use[B] ∪ (out[B] − def[B])` は伝達関数で、「`B` を逆向きに通ると、`B` で代入された変数は（その手前では）もう生きている必要がなくなり、`B` で読まれる変数が新たに必要になる」を表します。

## 不動点反復：方程式を繰り返しで解く

問題は、上の方程式が**循環している**ことです。`in[B]` は `out[B]` に依存し、`out[B]` は後続の `in` に依存し、ループがあれば巡り巡って自分に戻ってきます。普通には解けません。そこで使うのが **不動点反復（fixpoint iteration）** です。

考え方は素朴です。「全部を空集合（または全集合）から始め、方程式を**変化がなくなるまで**何度も適用し続ける」。値が増える（または減る）一方で、しかも上限があるので、有限回で必ず**変化しなくなる**（＝不動点に達する）。その状態が答えです。

```ruby
# 生存変数解析（後ろ向き）の不動点反復
def liveness(cfg)
  in_set  = Hash.new { |h, b| h[b] = Set.new }
  out_set = Hash.new { |h, b| h[b] = Set.new }

  changed = true
  while changed            # 変化がなくなるまで繰り返す
    changed = false
    cfg.blocks.reverse_each do |b|
      # out[B] = 後続の in の和
      new_out = b.successors.reduce(Set.new) { |acc, s| acc | in_set[s] }
      # in[B] = use[B] ∪ (out[B] − def[B])
      new_in  = b.use | (new_out - b.def)

      if new_in != in_set[b] || new_out != out_set[b]
        in_set[b]  = new_in
        out_set[b] = new_out
        changed = true     # まだ動いた → もう一周
      end
    end
  end

  [in_set, out_set]
end
```

このループが**必ず止まる**ことには、ちゃんとした理由があります。集合は単調に増える一方で、変数の総数という上限があるからです。この「単調 + 上限つき」という構造を一般化したのが、Kildall が1973年に与えた **格子（lattice）に基づくデータフロー解析の統一理論** です[Kildall, 1973](#cite:kildall1973)。

> [!NOTE]
> **格子（lattice）** とは、要素間に「より精密／より粗い」という順序が入り、任意の2要素に「両方を束ねた要素（上限）」が存在するような集合のことです。データフロー値の集まりは格子をなし、伝達関数が単調（情報を逆転させない）であれば、不動点反復は必ず一意の解（最小不動点）に収束します。この枠組みは次章「抽象解釈」へまっすぐつながります。Kildall の貢献は、個別の最適化ごとにバラバラだった解析を、この一つの数学的枠組みに統一したことにあります。

## ワークリスト：賢く反復する

「全ブロックを毎回なめ直す」のは無駄が多い。実際には、**変化があったブロックの隣だけ**を再計算すれば十分です。これを管理するのが **ワークリストアルゴリズム（worklist algorithm）** です。

```ruby
def liveness_worklist(cfg)
  in_set  = Hash.new { |h, b| h[b] = Set.new }
  out_set = Hash.new { |h, b| h[b] = Set.new }
  work = cfg.blocks.dup           # 最初は全ブロックが対象

  until work.empty?
    b = work.pop
    out_set[b] = b.successors.reduce(Set.new) { |a, s| a | in_set[s] }
    new_in = b.use | (out_set[b] - b.def)
    if new_in != in_set[b]
      in_set[b] = new_in
      work |= b.predecessors      # 変化したので前者だけを再投入
    end
  end
  [in_set, out_set]
end
```

ナイーブな全走査と結果は同じですが、必要なところだけ計算するので、ずっと速くなります。これが実用的なデータフロー解析エンジンの基本骨格です。前向きの解析（到達定義など）なら、たどる向きと `in`/`out` の役割を入れ替えるだけで、同じ枠組みがそのまま使えます。データフロー解析と次章の抽象解釈をまとめて体系的に学ぶには、Nielson らの教科書[Nielson et al., 1999](#cite:nielson1999)が定番です。

## 精度を上げる工夫

基本のデータフロー解析は強力ですが、精度には限界があります。それを押し上げる代表的な発展を3つ紹介します。いずれも実用処理系で広く使われています。

**SSA 形式（static single assignment）。** 「各変数への代入はプログラム中ただ1か所」という制約を満たすように、変数に番号を振り直した中間表現です。`x = 1; x = 2` を `x1 = 1; x2 = 2` と書き換え、分岐が合流する点では `x3 = φ(x1, x2)` という特別な **φ関数（phi function）** で「どちらの経路から来たかに応じた値」を表します。SSA にすると「ある使用に届く定義はただ1つ」が一目で分かり、定数伝播やデッドコード除去が劇的に簡単・高速になります。Cytron らが効率的な構築法（支配辺境を使う方法）を与えて以来[Cytron et al., 1991](#cite:cytron1991)、LLVM をはじめ現代の主要コンパイラの中核表現になっています。

**手続き間解析（interprocedural analysis）。** ここまでは1つの関数の中（手続き内）だけを見てきました。しかし「この関数を呼んだら戻り値は常に正か」を知るには、関数の**境界をまたいで**情報を流す必要があります。素朴にやると、呼び出し元と呼び出し先の対応（どの呼び出しに対応して戻るか）を取り違え、ありえない経路まで数えて精度が落ちます。Reps・Horwitz・Sagiv は、広い範囲の手続き間データフロー問題を**グラフ到達可能性問題**に変換し、呼び出しと戻りの対応を正しく保ったまま多項式時間で精密に解く方法（IFDS フレームワーク）を示しました[Reps et al., 1995](#cite:reps1995)。手続き間解析の理論的な背骨です。

**指示解析（points-to / pointer analysis）。** ポインタや参照がある言語では、「`p` が指す先は何か」が分からないと、`*p = 5`（C 風）や `obj.field = 5` が**どの変数・どのオブジェクトを変えるのか**確定できません。これが分からなければ、データフロー解析はあらゆる可能性を考えて精度を失います。そこで「各ポインタが指しうる対象の集合」を求めるのが指示解析です。Steensgaard は、精度をやや犠牲にする代わりに、ほぼ**線形時間**で巨大なプログラムも解ける手法（型推論の単一化に似た合併ベースの解析）を与えました[Steensgaard, 1996](#cite:steensgaard1996)。「速いが粗い」Steensgaard 流と「遅いが精密」な包含ベースの解析は、精度と速度のトレードオフの好対照としてよく引かれます。

> [!TIP]
> データフロー解析を読むときは、いつも「**向き**（前向きか後ろ向きか）」「**合流**（和か積か）」「**伝達関数**（何を足し何を消すか）」の3点を抜き出す癖をつけると、初めて見る解析でもすぐ構造が掴めます。生存変数は「後ろ向き・和・use を足し def を消す」、到達定義は「前向き・和・gen を足し kill を消す」、というように。

## まとめ

データフロー解析は、CFG の各地点でのある種の情報を、連立方程式の不動点として求める解析です。

- **動機**：値や情報の流れを知り、デッドコード除去・定数伝播などの最適化を可能にする（最適化、検証）。
- **結果**：各地点での生存変数・到達定義・定数性などの情報。
- **手法**：データフロー値・伝達関数・合流演算を決め、格子の上で不動点反復（ワークリスト）を回す。SSA・手続き間解析・指示解析で精度を高める。

ここで一つ、大きな問いが浮かびます。型推論もデータフロー解析も「制約を集めて不動点を求める」「安全側に近似する」点で似ていないか? ── その直感は正しく、両者は **抽象解釈** という一つの理論で説明できます。次章でその統一的な視点を手に入れましょう。
