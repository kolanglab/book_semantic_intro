# 型検査

名前が解決できたら、次に問いたいのは「この式は**型として筋が通っているか**」です。`1 + 2` は問題なくても `1 + "two"` はあやしい。こうした判断を実行前に下すのが **型検査（type checking）** です。型は意味解析のなかでもとりわけ理論が整っていて、しかも実用的な見返りが大きい分野です。

## 型は何を約束するのか

**型（type）** とは、値の集まりに付けた「種類」のラベルです。`Integer` は整数全体、`String` は文字列全体、`Integer -> Integer` は「整数を受け取り整数を返す関数」全体、というように。型検査の目的は、プログラム中のすべての式に矛盾なく型を割り当てられるかを確かめ、割り当てられないなら**型エラー**として報告することです。

なぜそんなことをするのか。型が通っている（**well-typed** な）プログラムには、ある種の実行時エラーが**起きないことが保証される**からです。Milner はこれを「型の付くプログラムは（型に関して）おかしくならない（*well-typed programs cannot go wrong*）」という有名な標語で表現しました[Milner, 1978](#cite:milner1978)。たとえば静的型のある言語では、「整数に対して文字列専用のメソッドを呼ぶ」「関数でないものを呼び出す」といった誤りが、コンパイル時に弾かれます。

型検査の動機を整理すると、

- **検証**：型の合わない演算・呼び出しを実行前に発見する。バグの早期発見であり、いわば「機械が証明してくれる軽量なテスト」です。
- **情報付与**：各式の型が分かると、コード生成で「整数加算命令か浮動小数点加算命令か」を選べる、メモリ上のサイズが決まる、といった具体的な利益があります。

> [!NOTE]
> 「型がある言語」と「型検査が静的な言語」は別の話です。Ruby や Python にも値の型はありますが、検査は主に**実行時**に行われます（動的型付け）。本章で扱うのは、実行前に AST から型を調べる**静的型検査**です。なお近年は Ruby の RBS / Steep や Python の型ヒント + mypy のように、動的型言語に静的型検査を**後付け**する流れもあります。

## 型システムと型規則

型をどう割り当てるかのルール全体を **型システム（type system）** と呼びます。型システムは、「どんな式にどんな条件でどんな型が付くか」を **型規則（typing rule）** の集まりとして定めます。型規則は伝統的に、横線の上に「前提」、下に「結論」を書く形式で表します。

たとえば「整数同士の足し算は整数」という規則は、こう書けます。

```text
   Γ ⊢ e1 : Int     Γ ⊢ e2 : Int
  ───────────────────────────────── (T-Add)
        Γ ⊢ e1 + e2 : Int
```

記号の読み方を説明します。`Γ`（ガンマ）は **型環境（type environment）**、すなわち「いま見えている変数とその型の対応表」です（名前解決で作ったシンボルテーブルの型版だと思ってください）。`Γ ⊢ e : T` は「型環境 Γ のもとで、式 e は型 T を持つ」と読みます。この規則 (T-Add) は、「`e1` が `Int`、`e2` が `Int` ならば、`e1 + e2` は `Int`」と言っているだけです。横線の上が成り立てば下が成り立つ、という含意です。

他の規則も同じ調子で書けます。

```text
   (x : T) ∈ Γ                 Γ ⊢ c : 整数リテラルなら Int
  ───────────── (T-Var)       ──────────────────────────── (T-Int)
   Γ ⊢ x : T                       Γ ⊢ 42 : Int


   Γ ⊢ cond : Bool    Γ ⊢ e1 : T    Γ ⊢ e2 : T
  ────────────────────────────────────────────── (T-If)
            Γ ⊢ if cond then e1 else e2 : T
```

(T-Var) は「環境に `x : T` という登録があれば、`x` の型は `T`」。(T-If) は「条件は `Bool` で、then 節と else 節が**同じ型** `T` なら、if 式全体の型は `T`」です。if の両腕が同じ型でなければならないのは、「式としての if がどちらに転んでも同じ型の値を返す」ことを保証するためです。

型規則は一見もったいぶって見えますが、利点があります。**実装と仕様を分離できる**のです。「言語の型はこうあるべき」を型規則として正確に書き、処理系はそれを忠実に実装すればよい。曖昧な日本語仕様より、はるかに誤解が少なくなります。型システムの体系的な入門としては[Pierce, 2002](#cite:pierce2002)が定番です。

## 型検査を実装する

型規則は、ほとんどそのまま AST のビジターとして実装できます。型規則の「横線の上から下へ」は、子の型から親の型を組み立てる **合成属性**（下から上）に対応します。前章のスコープが上から下だったのと、ちょうど逆向きです。

```ruby
class TypeChecker < Visitor
  def initialize(env)
    @env = env   # 型環境 Γ：変数名 => 型
  end

  # 整数リテラル → Int    (T-Int)
  def visit_int(node)
    :Int
  end

  # 変数参照 → 環境に登録された型    (T-Var)
  def visit_var(node)
    @env[node.name] or raise "未定義の変数: #{node.name}"
  end

  # 二項演算 + → 両辺が Int なら Int    (T-Add)
  def visit_add(node)
    lt = visit(node.left)
    rt = visit(node.right)
    unless lt == :Int && rt == :Int
      raise "型エラー: Int + Int を期待したが #{lt} + #{rt}"
    end
    :Int
  end

  # if 式 → 条件は Bool、両腕は同じ型    (T-If)
  def visit_if(node)
    raise "条件は Bool が必要" unless visit(node.cond) == :Bool
    t1 = visit(node.then)
    t2 = visit(node.else)
    raise "if の両腕の型が不一致: #{t1} と #{t2}" unless t1 == t2
    t1
  end
end
```

`visit` の戻り値がその式の型です。子を先に `visit` して型を求め（帰りがけ）、それを使って自分の型を決める ── これが「型は下から上へ流れる合成属性」の正体です。型エラーは、規則の前提が満たされない瞬間に検出されます。たとえば `visit_add` で `lt` が `:String` だったら、その場で型エラーになります。

**結果**として得られるのは、(1) 型エラーがあれば具体的な場所と内容の報告、(2) エラーがなければ、各 AST ノードに型が注釈された装飾済み AST、です。この型注釈は後続のコード生成で命令選択などに使われます。

## 型の健全性 ── 検査は何を保証するのか

型検査が「型が通った」と言ったとき、それは本当に安全なのでしょうか。この問いに答えるのが **型の健全性（type soundness）** です。健全性はふつう、次の2つの補題に分けて証明されます。これは Wright と Felleisen による **構文的アプローチ（syntactic approach）** として知られる、いまや標準的な手法です[Wright and Felleisen, 1994](#cite:wright1994)。

- **進行（progress）**：型の付いた式は、すでに値であるか、さもなくば必ず一歩実行を進められる。つまり「型は通っているのに次に何をすればいいか分からず固まる（stuck する）」ことがない。
- **保存（preservation, subject reduction）**：型の付いた式を一歩実行しても、型は変わらない（保たれる）。

この2つを合わせると、「型の付いたプログラムは、stuck することなく実行が進み、しかも型はずっと保たれる」、すなわち**型エラーで異常停止しない**ことが言えます。これが Milner の標語の中身です。

> [!IMPORTANT]
> 健全性は「型が通れば**あらゆる**バグがない」という意味ではありません。型システムが対象とする種類のエラー（型の不整合）が起きない、という保証です。配列の範囲外アクセスやゼロ除算は、ふつうの型システムの対象外なので、型が通っても起こりえます。型が何を保証**しない**かを正しく知ることも、処理系を理解するうえで大切です。

逆に、健全でない（unsound な）型システムも現実には存在します。実用上の利便性のために、わざと健全性を犠牲にした「抜け穴」を持つ言語もあります（たとえば配列の共変性をめぐる古典的な例）。「健全性」と「使いやすさ・表現力」のあいだのトレードオフは、型システム設計の永遠のテーマです。

## サブタイピング：型に「広い・狭い」を入れる

ここまでは型の一致を `==` で見てきましたが、実際の言語ではもう少し柔軟です。「`Cat` は `Animal` の一種だから、`Animal` を要求する場所に `Cat` を渡せる」といった関係を扱いたい。これを **サブタイピング（subtyping, 部分型付け）** と呼びます。`Cat` が `Animal` の部分型であることを `Cat <: Animal` と書きます。

サブタイピングの心は **置換可能性（substitutability）**：「`T` が要求される場所には、`T` の部分型 `S` の値を置いてよい」という規則です。これは次の **包摂規則（subsumption rule）** で表されます。

```text
   Γ ⊢ e : S      S <: T
  ──────────────────────── (T-Sub)
         Γ ⊢ e : T
```

「`e` が `S` 型で、`S` が `T` の部分型なら、`e` を `T` 型とみなしてよい」。これにより、`Animal` 型の引数に `Cat` 型の値を渡せるようになります。型検査の `==` を「部分型か（`<:`）」の判定に置き換えるのが、サブタイピング対応の第一歩です。クラス継承を持つオブジェクト指向言語の型検査は、本質的にこの部分型関係の判定に支えられています。

> [!TIP]
> サブタイピングは便利ですが、規則を増やすと健全性の証明が一気に難しくなります。とくに関数型や可変参照とサブタイピングの相互作用（共変・反変）は誤りの温床です。「引数の型は反変、戻り値の型は共変」という関数型のサブタイピング規則は、最初は直感に反するかもしれませんが、置換可能性から論理的に導かれます。

## まとめ

型検査は、型システムの定める型規則に従って各式に型を割り当て、矛盾があれば型エラーとして報告する解析です。

- **動機**：型の不整合を実行前に検出する（検証）。各式の型を確定させる（情報付与）。
- **結果**：型エラーの報告、または各ノードに型が付いた装飾済み AST。
- **手法**：型規則を合成属性（下から上）としてビジターで実装する。サブタイピングがあれば一致判定を部分型判定に置き換える。
- **保証**：健全な型システムなら、進行と保存により「型の付いたプログラムは型エラーで止まらない」ことが言える。

ここまでは、プログラマが型を**書いてくれる**前提でした。では、型注釈が一切なくても処理系が型を**自分で推測**できるでしょうか。次章「型推論」で、その鮮やかな仕組みを見ます。
