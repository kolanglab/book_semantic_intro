# 型推論

前章では、プログラマが型注釈を書いてくれる前提で型検査をしました。しかし OCaml や Standard ML、Haskell では、ほとんど型を書かないのに処理系が正確な型を割り出してくれます。この「注釈なしで型を割り出す」解析が **型推論（type inference）** です。本章ではその古典かつ中核である **ヒンドリー・ミルナー型推論（Hindley-Milner type inference, HM）** を、なるべく具体的に追います。なお、Rust や TypeScript も多くの型を自動で推測しますが、所有権・トレイト・サブタイピング・union 型・制御フローによる型の絞り込みなどが絡む別種の設計で、古典的な HM そのものではありません（章末で再訪します）。

## 動機：型は書きたくないが恩恵は受けたい

静的型の安心は欲しい。でも、すべての変数と関数にいちいち型を書くのは面倒だし、コードが読みにくくなる。この2つを両立させるのが型推論です。次の OCaml 風コードを見てください。

```ocaml
let double f x = f (f x)
```

型注釈は一つもありません。それでも処理系は、`double` の型が

```text
('a -> 'a) -> 'a -> 'a
```

だと推論できます。読み方は「『ある型 `'a` から同じ `'a` への関数』と、『`'a` の値』を受け取り、`'a` を返す」。ここで `'a` は **型変数（type variable）**、つまり「どんな型でもよい」プレースホルダです。`double` は整数にも文字列にも使えるので、特定の型に固定せず**多相的（polymorphic）** な型が付きます。これを処理系が**自動で**、しかも**一番一般的な型**（後述の主要型）として導けるのが HM の凄みです。

型推論の理論的な出発点は Hindley の主要型に関する研究[](#cite:hindley1969)で、それをプログラミング言語に持ち込んで体系化したのが Milner[](#cite:milner1978)、そして実用的なアルゴリズムとして整理したのが Damas と Milner です[](#cite:damas1982)。

## 考え方：未知数を置いて、制約を解く

HM の発想は、中学校で習う連立方程式とそっくりです。

1. 型が分からないところには、ひとまず **型変数**（未知数）を置く。
2. AST をたどりながら、「ここはこの型と等しくないとおかしい」という **制約（constraint, 等式）** を集める。
3. 集めた制約を **単一化（unification）** で解き、型変数の正体を突き止める。

たとえば `f x` という関数適用を見たら、「`f` は『`x` の型を受け取って何かを返す関数』のはずだ」という制約が立ちます。`f` の型を `t_f`、`x` の型を `t_x`、適用全体の型を `t_r` と置くと、

```text
   t_f = t_x -> t_r
```

という等式が得られます。プログラム全体からこうした等式を集め、まとめて解くのです。

## 単一化：型の等式を解く

**単一化（unification）** は、2つの型を「等しくするには型変数に何を当てはめればよいか」を求める操作です。求めた当てはめを **代入（substitution）** と呼びます。規則は驚くほど単純です。

- 型変数 `'a` と型 `T` を単一化 → `'a` を `T` とする（ただし `'a` が `T` の中に現れてはいけない。これを **occurs check** と呼ぶ）。
- `A -> B` と `C -> D` を単一化 → `A` と `C`、`B` と `D` をそれぞれ単一化する。
- 同じ基本型同士（`Int` と `Int`）→ 成功、何もしない。
- それ以外（`Int` と `String`、`Int` と `A -> B` など）→ **型エラー**。

Ruby で書くと、その素朴さがよく分かります。

```ruby
# 型の表現：型変数 Var、基本型 Base、関数型 Fun
Var  = Struct.new(:name)            # 'a, 'b, ...
Base = Struct.new(:name)            # Int, Bool, ...
Fun  = Struct.new(:arg, :ret)       # arg -> ret

class Unifier
  def initialize
    @subst = {}    # 型変数名 => その正体（代入）
  end

  # t1 と t2 を等しくするよう代入を更新する
  def unify(t1, t2)
    t1 = resolve(t1)
    t2 = resolve(t2)

    if t1.is_a?(Var)
      bind(t1, t2)
    elsif t2.is_a?(Var)
      bind(t2, t1)
    elsif t1.is_a?(Fun) && t2.is_a?(Fun)
      unify(t1.arg, t2.arg)         # 引数同士
      unify(t1.ret, t2.ret)         # 戻り値同士
    elsif t1.is_a?(Base) && t2.is_a?(Base) && t1.name == t2.name
      # 同じ基本型：OK
    else
      raise "型エラー: #{show(t1)} と #{show(t2)} は単一化できません"
    end
  end

  private

  # すでに判明している代入を最後まで適用する
  def resolve(t)
    while t.is_a?(Var) && @subst.key?(t.name)
      t = @subst[t.name]
    end
    t
  end

  def bind(var, t)
    return if t.is_a?(Var) && t.name == var.name   # 'a = 'a は何もしない
    raise "occurs check 失敗" if occurs?(var, t)    # 'a = 'a -> 'b を防ぐ
    @subst[var.name] = t
  end

  def occurs?(var, t)
    t = resolve(t)
    case t
    when Var then t.name == var.name
    when Fun then occurs?(var, t.arg) || occurs?(var, t.ret)
    else false
    end
  end

  # 型エラーのメッセージ用に、型を読みやすい文字列にする
  def show(t)
    t = resolve(t)
    case t
    when Var  then t.name
    when Base then t.name.to_s
    when Fun  then "(#{show(t.arg)} -> #{show(t.ret)})"
    end
  end
end
```

`occurs check` は何のためにあるのでしょうか。`'a = 'a -> 'b` のような循環した等式を許すと、`'a` が自分自身を含む**無限に大きな型（再帰型）**になってしまいます。通常の HM 型システムはこうした無限型を扱わないため、循環する代入は型エラーとして拒否する ── それが occurs check です（再帰型を明示的に扱う型システムや、循環する項を許す単一化を使う実装では、別の扱いになります）。地味ですが、HM の型推論に欠かせない安全弁です。

## 制約を集めながら木をたどる

単一化器ができたら、AST をたどって制約（`unify` の呼び出し）を発行していきます。各式に対して「新しい型変数を割り当て、子の型との関係を `unify` で表明する」のが基本パターンです。

```ruby
class Inferencer < Visitor
  def initialize(unifier)
    @u   = unifier
    @env = {}   # 変数名 => 型
    @cnt = 0
  end

  def fresh = Var.new("t#{@cnt += 1}")   # 新しい型変数を作る

  def visit_int(node)  = Base.new(:Int)
  def visit_bool(node) = Base.new(:Bool)

  def visit_var(node)
    @env.fetch(node.name) { raise "未定義: #{node.name}" }
  end

  # 関数適用 f(x)：f の型は「x の型 -> 結果の型」でなければならない
  def visit_app(node)
    tf = visit(node.fun)
    tx = visit(node.arg)
    tr = fresh
    @u.unify(tf, Fun.new(tx, tr))   # ← ここで制約を発行
    tr
  end

  # 無名関数 fun x -> body：引数に新しい型変数を割り当てて本体を見る
  def visit_lambda(node)
    tx = fresh
    @env[node.param] = tx
    tbody = visit(node.body)
    Fun.new(tx, tbody)
  end
end
```

`visit_app` の `@u.unify(tf, Fun.new(tx, tr))` が、まさに「`f` は『`x` の型 → 結果の型』のはず」という制約です。木をたどり終えると、`@u` の中の代入に型変数の正体がすべて溜まっています。最後に各式の型に代入を適用すれば、推論された型の出来上がりです。

たとえば `(fun x -> x + 1)` を推論すると、`x` に型変数 `t1` を割り当て、`x + 1` の足し算から `t1 = Int` という制約が立ち、単一化の結果 `t1` が `Int` に確定します。よって全体の型は `Int -> Int`。**注釈ゼロから型が決まりました。**

## 主要型と let 多相

HM の理論的な美点は、「型が付くなら、**いちばん一般的な型**（**主要型／principal type**）がただ一つ存在し、それを必ず見つけられる」ことです[](#cite:damas1982)。冒頭の `double` に `('a -> 'a) -> 'a -> 'a` という最も一般的な型が付いたのは、この性質のおかげです。「一番一般的」とは、他のどの正しい型もそれの特殊化（型変数を埋めたもの）になっている、という意味です。また「ただ一つ」といっても、型変数の名前の違いは同一視します。`'a -> 'a` と `'b -> 'b` は、変数名を付け替えただけの同じ主要型です。

ここで一つ重要な仕掛けが **let 多相（let-polymorphism, 全称化／generalization）** です。`let` で名前に束縛された値の型に残った型変数は、「どんな型にもなれる」型として**一般化**され、使うたびに別の型として具体化（**インスタンス化／instantiation**）できます。

```ocaml
let id = fun x -> x in   (* id : 'a -> 'a と一般化される *)
let a = id 3 in          (* ここでは 'a = Int として使う  *)
let b = id "hi" in       (* ここでは 'a = String として使う *)
...
```

`id` は一度定義しただけなのに、整数にも文字列にも使えています。一般化のおかげで `'a -> 'a` という「型の設計図（**型スキーム／type scheme**）」になり、使うたびに新鮮な型変数として焼き直されるからです。これがなければ、`id 3` を見た瞬間に `'a` が `Int` に固定され、`id "hi"` が型エラーになってしまいます。let 多相は、再利用可能な多相関数を自然に書けるようにする、HM の要です。

> [!NOTE]
> 一般化はどこでもしてよいわけではありません。関数の引数に取った変数（ラムダ束縛）は一般化**しません**。`fun id -> (id 3, id "hi")` は型が付かないのが正解です。「let で束縛したものだけ一般化する」という線引きが、健全性と推論可能性のちょうどよい妥協点になっています。また let でも、推論された型のすべての型変数を一般化するのではなく、**現在の型環境に自由に現れていない**型変数だけを一般化します。外側から来た型変数まで一般化すると、外側の仮定との関係が壊れてしまうからです。さらに ML 系言語では、可変参照のような**副作用を持つ式**を多相的に一般化すると不健全になるため、原則として「値」の形をした式だけを一般化する **value restriction（値制限）** という制限が入ります（OCaml の relaxed value restriction のように、もう少し柔軟に一般化する設計もあります）。

## HM の限界とその先

HM は「注釈なしで主要型を必ず見つける」という、理論と実用のまれな両立を達成しました。だからこそ40年以上たった今も、多くの言語の型推論の土台になっています。一方で限界もあります。

- **完全な推論をあきらめる代わりに表現力を上げる**：サブタイピング、多相の多相（高階多相、System F 相当）、依存型などを足すと、一般に型推論は**判定不能**になります。そこで現実の言語は「ときどき型注釈を要求する」ことで折り合いをつけます。Rust や TypeScript の「局所的には推論、関数の境界では注釈」という設計はその一例です。
- **エラーメッセージが分かりにくい**：制約をまとめて解くため、型エラーが「最初に矛盾した場所」ではなく「矛盾が露見した場所」で報告されがちです。エラーの分かりやすさは、いまも活発な研究テーマです。

そして HM の「型変数に値を当てはめて制約を解く」という発想自体は、第4部で扱う**篩型（refinement types）** ── 型に「正でなければならない」といった述語を載せ、その制約を SMT ソルバで解く ── へと自然に発展していきます。型推論は、最先端の検証技術へ続く入口でもあるのです。

## まとめ

型推論は、型注釈なしのプログラムから型を自動的に導く解析です。

- **動機**：静的型の安全性を、型を書く手間なしに享受する（情報付与）。
- **結果**：注釈ゼロでも、各式に主要型（最も一般的な型）が付いた装飾済み AST。
- **手法**：未知の型に型変数を置き、AST から型の等式（制約）を集め、単一化で解く。`let` 束縛では一般化して多相を実現する。

名前と型 ── プログラムの「構造」に関する解析はここまでです。第3部からは視点を変え、プログラムを実行したときに制御と値が**どう流れるか**を、実行せずに追いかけます。
